Middleware ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

Middleware in ASP.NET Core is software that is assembled into an application pipeline to handle requests and responses. Each component chooses whether to pass the request to the next component in the pipeline, and can perform work before and after the next component is invoked in the pipeline.

30-Second Elevator Pitch
Perfect Answer

Middleware is a code component injected into the HTTP request/response pipeline. In ASP.NET Core, it's configured in the Program.cs file. Each middleware acts on the incoming HTTP Request, optionally calls the next middleware using 'next()', and processes the outgoing HTTP Response in reverse order. Common built-in examples are Routing, Authentication, and Static Files.

Deep-Dive Answer (2 Minutes)

Middleware functions as a chain of responsibility. When a request hits the Kestrel web server, it passes through a pipeline of middleware components. Each component can intercept the request, perform logic (e.g., logging, validation), and either short-circuit the pipeline (e.g., if authentication fails) or pass control to the next delegate using the RequestDelegate. When the endpoint (like an API Controller) finishes processing, the response travels backward through the same pipeline, allowing middlewares to inspect or modify the response headers and body before it goes back to the client. We configure this in Program.cs using extension methods like UseRouting(), UseAuthentication(), and custom middleware classes.

Real-Time Metaphor

A security gate check at an airport. Every passenger (request) must pass through identity verification, baggage scanning, and boarding pass checks (individual middlewares). If any check fails (short-circuit), they are turned back (response returned). If all pass, they board the plane (endpoint controller). On the way back, bags are reclaimed in reverse order.

Real Project Implementation

In our e-commerce API, we wrote a custom CorrelationIdMiddleware. For every incoming request, it checks if a 'X-Correlation-ID' header exists. If not, it generates a new GUID and adds it. It pushes this ID into the Serilog diagnostic context, calls next(), and adds the header to the response. This ensures all logs across microservices for a single user action share the same ID.

Internal Mechanics & Working

ASP.NET Core represents middleware using the RequestDelegate, which is a delegate representing a method that takes a HttpContext and returns a Task: `public delegate Task RequestDelegate(HttpContext context)`. In Program.cs, when we call `app.Use()`, we pass a function that takes the `HttpContext` and the next `RequestDelegate` in the chain. The builder chains them together, returning a single RequestDelegate representing the entire pipeline. The host server executes this delegate on every HTTP request.

Advantages & Disadvantages
Advantages

1. Extremely lightweight and fast compared to old ASP.NET HTTP Modules. 2. Highly configurable: you only add what you need. 3. Clean Separation of Concerns; cross-cutting concerns (logging, auth, error handling) are kept out of business controllers.

Disadvantages

1. Execution order is critical; a misplaced middleware (e.g., calling UseAuthorization before UseRouting) causes runtime failures or security vulnerabilities. 2. Debugging across multiple nested middlewares can sometimes be difficult.

Best Practices
Architect Tips

1. Always define custom middlewares in separate classes and expose them via clean extension methods like `app.UseMyCustomMiddleware()`. 2. Short-circuit the pipeline early (e.g., rate-limiting, auth) to avoid unnecessary processing. 3. Inject scoped services into the InvokeAsync/Invoke method, NOT the constructor, to avoid singleton lifetime captive dependency issues.

Code Example
C# Source 
public class CorrelationIdMiddleware
{
    private readonly RequestDelegate _next;
    private const string CorrelationIdHeader = "X-Correlation-ID";

    public CorrelationIdMiddleware(RequestDelegate next)
    {
        _next = next;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        if (!context.Request.Headers.TryGetValue(CorrelationIdHeader, out var correlationId))
        {
            correlationId = Guid.NewGuid().ToString();
        }

        context.Items[CorrelationIdHeader] = correlationId.ToString();
        context.Response.Headers.Add(CorrelationIdHeader, correlationId);

        await _next(context);
    }
}
Common Developer Pitfalls
Mistake Alert

Registering a scoped service inside the middleware constructor. Since Middleware is registered as a Singleton, injecting a Scoped DbContext into the constructor causes memory leaks and DbContext multi-thread access errors. Always inject scoped dependencies directly into the InvokeAsync method signature.

Follow-ups:

  • How do you pass data from one middleware to another?
Advanced:
  • How does ASP.NET Core resolve capturing scoped dependencies in a singleton middleware?
  • How does app.Map branch the request pipeline internally?
Interview Advice

Tips: If asked about middleware, immediately mention that order is key. Draw the pipeline diagram on a whiteboard if available. Mention the difference between constructor injection (singleton) and method injection (scoped).

Memory Trick: Middleware is like a 'Chain of filters' in a water purification system. Water goes through sediment, charcoal, and UV filters in a specific order. If any filter detects severe contamination, it stops the flow.

Company Expectations: The interviewer wants to see if you understand the RequestDelegate, the lifetime of middleware components, and how to avoid the 'captive dependency' bug.

Middleware Workflow ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

The Middleware Workflow is the sequence and directional flow of execution that HTTP requests and responses take as they traverse the ASP.NET Core pipeline, guided by the Use, Run, and Map pipeline configurations.

30-Second Elevator Pitch
Perfect Answer

The workflow is bidirectional: requests flow in, get processed, reach an endpoint, and responses flow back in reverse order. It is configured in Program.cs. `app.Use()` adds a middleware that can call the next or short-circuit, `app.Run()` terminates the pipeline, and `app.Map()` branches it.

Deep-Dive Answer (2 Minutes)

The workflow begins when Kestrel creates a HttpContext for an incoming TCP request. The context is passed to the first middleware delegate. The middleware performs its 'pre-processing' logic, then calls `await next.Invoke(context)` to delegate control forward. This continues until a terminal middleware (like app.Run or the EndpointMiddleware) processes the request and generates a response without calling next. At that point, the call stack unwinds. Control flows back through each middleware in reverse order, executing code after the `await next` call ('post-processing'). Common order is: Exception Handler -> CORS -> Static Files -> Routing -> Authentication -> Authorization -> Endpoint.

Real-Time Metaphor

A package delivery through postal checkpoints. The package goes from Checkpoint A to B to C (pre-processing). At C, the package is delivered (endpoint). The delivery receipt is signed and travels back from C to B to A (post-processing) so everyone knows it succeeded.

Real Project Implementation

In our ASP.NET Core API, we use ExceptionHandlingMiddleware as the absolute first middleware. If any downstream middleware (like Authentication) or controller throws an exception, it bubbles up to the ExceptionHandlingMiddleware, which catches it and returns a clean, standardized JSON response to the user instead of letting the application crash.

Internal Mechanics & Working

Each middleware registers a delegate of type `Func`. During startup, ASP.NET Core uses `IApplicationBuilder` to compose these functions. It calls them in reverse registration order, passing each middleware's output delegate to the previous one as its 'next' parameter. The result is a single nested delegate. If `next()` is omitted in any middleware, downstream middlewares are never executed, short-circuiting the flow.

Advantages & Disadvantages
Advantages

1. Allows request filtering early, preventing heavy business logic execution if pre-conditions fail. 2. Modular pipeline architecture keeps boot times incredibly fast. 3. Exception handling is centralized at the root of the workflow.

Disadvantages

1. If a middleware forgets to call `await next()`, the request hangs or returns blank, which can be hard to track. 2. Writing async void middlewares will break the workflow, causing unhandled thread issues.

Best Practices
Architect Tips

1. Always await the `next(context)` call to avoid blocking threads. 2. Place global exception handling and logging middlewares at the very beginning of the pipeline. 3. Use `app.Map()` for branching paths like '/health' checks to bypass security middleware.

Code Example
C# Source 
app.Use(async (context, next) => {
    // Pre-processing
    Console.WriteLine("Request path: " + context.Request.Path);
    
    await next(); // Pass control to next middleware
    
    // Post-processing
    Console.WriteLine("Response status: " + context.Response.StatusCode);
});
Common Developer Pitfalls
Mistake Alert

Calling `context.Response.WriteAsync()` in a middleware pre-processing step and then calling `await next()`. Once the response body starts writing, headers are locked. Downstream middlewares will fail when they try to modify headers (like cookies or authentication).

Follow-up & Advanced Questions
Follow-ups:
  • What happens if you don't await the next() call?
  • Can we dynamically skip a middleware during runtime?
Advanced:
  • How is HttpContext scoped across async calls in the middleware workflow?
  • Explain how the developer exception page middleware works internally compared to production exception handling.
Interview Advice

Tips: Emphasize that the Exception Handler is placed first so it wraps the entire execution block, catching exceptions from all subsequent middlewares.

Memory Trick: Think of nested Russian Dolls. The request goes from the outermost doll to the innermost. The response travels from the innermost doll back to the outermost.

Company Expectations: The company wants to see if you can debug a broken request pipeline and if you know why placing Authorization before Authentication is a critical bug.

Abstract Class ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

An abstract class in C# is a class declared with the 'abstract' keyword. It cannot be instantiated directly and serves as a blueprint for other classes. It can contain abstract methods (which have no implementation and must be overridden) as well as concrete methods (which have implementations).

30-Second Elevator Pitch
Perfect Answer

An abstract class is a restricted class that cannot be used to create objects. It is designed to be inherited. It allows us to define shared behavior (concrete methods) and force child classes to implement specific logic (abstract methods). It supports fields, properties, constructors, and access modifiers.

Deep-Dive Answer (2 Minutes)

Abstract classes provide a common base definition of a class that multiple derived classes can share. Unlike interfaces, abstract classes can hold state (fields), have constructors, define access modifiers (protected, internal, private), and implement concrete methods. They are used to implement the 'is-a' relationship. Any class inheriting an abstract class must implement all abstract methods unless the child class is also abstract. You define abstract methods with no body, and derived classes override them using the 'override' keyword.

Real-Time Metaphor

A generic 'TaxCalculator'. It has common methods like 'LogCalculation()' and shared fields like 'TaxYear'. However, the actual rate calculation is different for each country. So, the method 'CalculateTax()' is marked abstract. Children like 'USTaxCalculator' and 'UKTaxCalculator' provide concrete implementations.

Real Project Implementation

In our banking application, we have an abstract class `BasePaymentGateway`. It contains properties like `GatewayUrl` and `ApiKey`, concrete methods like `LogTransaction()`, and abstract methods like `ProcessPayment()` and `RefundPayment()`. Gateways like `StripeGateway` and `PayPalGateway` inherit this class and implement their own payment/refund API calls.

Visual Workflow
Abstract Base: BasePaymentGateway
Defines: LogTransaction() [Concrete] & ProcessPayment() [Abstract]
Child Class: StripeGateway
Overrides ProcessPayment() using Stripe SDK
Child Class: PayPalGateway
Overrides ProcessPayment() using PayPal API
Internal Mechanics & Working

At compilation, the abstract class is marked in MSIL (Microsoft Intermediate Language) with the `abstract` flag. The CLR blocks any attempt to call `new` on it. It supports polymorphic dispatch: when a method is called on a reference of the abstract base class, the CLR looks up the virtual method table (vtable) of the actual runtime object to execute the child's overridden method.

Advantages & Disadvantages
Advantages

1. High code reusability through inheritance of concrete methods and state. 2. Enforces a consistent contract across derived classes. 3. Allows defining constructors to initialize base state.

Disadvantages

1. C# does not support multiple inheritance; a class can inherit only one abstract class. 2. Tight coupling: changing the abstract class can break all derived subclasses (Fragile Base Class problem).

Best Practices
Architect Tips

1. Use abstract classes when derived classes share a lot of common implementation. 2. Keep abstract classes open for extension but closed for modification (SOLID OCP). 3. Make constructors protected, as they should only be called from derived classes.

Code Example
C# Source 
public abstract class Account
{
    public string AccountNumber { get; set; }
    public decimal Balance { get; protected set; }

    protected Account(string accNum)
    {
        AccountNumber = accNum;
    }

    public void Deposit(decimal amount)
    {
        Balance += amount;
    }

    public abstract void Withdraw(decimal amount); // Must implement
}
Common Developer Pitfalls
Mistake Alert

Trying to make an abstract method private. Abstract methods must be visible to child classes to be overridden, so they cannot be private (they must be public or protected).

Follow-up & Advanced Questions
Follow-ups:
  • Can an abstract class have a constructor? Yes, to initialize base properties.
  • Can an abstract class be static? No, static classes cannot be inherited.
Advanced:
  • How does the compiler handle virtual method tables (vtables) for abstract classes?
  • Can you declare an abstract class without any abstract methods? Yes, to prevent instantiation.
Interview Advice

Tips: Mention that abstract classes represent IS-A relationships (e.g., SavingsAccount IS-A Account), whereas interfaces represent CAN-DO capabilities (e.g., ISerializable can serialize).

Memory Trick: An abstract class is like a house blueprint. You can't live in a blueprint (can't instantiate it), but it contains pre-designed rooms (concrete methods) and empty spaces left for you to customize (abstract methods).

Company Expectations: The interviewer expects you to know that abstract classes allow shared code and state, which interfaces (prior to C# 8) could not do.

Explain Liskov Substitution Principle ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

The Liskov Substitution Principle (LSP) is the 'L' in SOLID. It states that objects of a superclass should be replaceable with objects of its subclasses without breaking the application. In other words, a derived class must not alter the expected behavior of the base class.

30-Second Elevator Pitch
Perfect Answer

Liskov Substitution Principle says that if class B inherits from class A, we should be able to replace A with B anywhere in our code without changing the correctness of the program. A subclass should not throw NotImplementedException for base methods or weaken postconditions.

Deep-Dive Answer (2 Minutes)

LSP is about behavioral subtyping. It enforces that inheritance is not just about sharing method signatures, but respecting the contract established by the base class. If a parent class promises a certain behavior (e.g., returning positive numbers, or completing a process without errors), the child class must honor that promise. Classic violations of LSP include throwing `NotImplementedException` in a child class because a base method isn't applicable, or writing conditional checks like `if (obj is DerivedClass)` to bypass normal execution. To follow LSP, subclasses must only extend base capabilities, not restrict or break them.

Real-Time Metaphor

The classic Rectangle and Square problem. If a Rectangle class has virtual properties for Width and Height, and a Square class overrides them so that setting Width also changes Height, a method expecting a Rectangle (which assumes setting Width does not alter Height) will calculate incorrect areas. Square is not a substitute for Rectangle here.

Real Project Implementation

In our notification service, we have an abstract class `NotificationSender` with a method `Send(string message, string recipient)`. We had a child class `SmsSender` which threw a `NotImplementedException` if recipient was an email address, or if message was too long. This violated LSP. We refactored it by introducing separate `EmailRecipient` and `PhoneRecipient` types and validating input before calling polymorphic methods.

Visual Workflow
Base Interface: IWritableFile
Promise: Can write byte data to storage
Good Child: LocalFile
Writes data to local disk (Success)
LSP Violator: ReadOnlyFile
Throws WriteDeniedException! (Breaks contract, client crashes)
Internal Mechanics & Working

LSP relates directly to Covariance and Contravariance in method signatures. A derived class method signature must be contravariant in arguments (accept at least what the base accepted) and covariant in return types (return a subtype of the base return). The CLR runtime handles types polymorphically, but LSP violation is a design-time logic bug, not a CLR runtime crash.

Advantages & Disadvantages
Advantages

1. Enables robust polymorphism and prevents unexpected runtime crashes. 2. Encourages loose coupling and modularity. 3. Eliminates the need for 'type checking' (is, as casting) code blocks.

Disadvantages

1. Requires careful, upfront design of object hierarchies. 2. Can lead to creating more interfaces or classes to split mismatched behaviors.

Best Practices
Architect Tips

1. Do not inherit a class just to reuse a single method; use Composition over Inheritance if you don't support the full contract. 2. Never throw `NotImplementedException` in derived implementations of virtual/abstract methods. 3. Keep interfaces small and focused (Interface Segregation) to avoid forcing subclasses to implement irrelevant methods.

Code Example
C# Source 
// LSP Violation Example
public class Customer
{
    public virtual void MakePayment() { /* process */ }
}

public class InquiryCustomer : Customer
{
    public override void MakePayment() 
    {
        throw new NotImplementedException("Inquiry customers cannot make payments!"); // LSP Violated!
    }
}
Common Developer Pitfalls
Mistake Alert

Overriding a base property or method to restrict its behavior, such as validating parameters in the child class more strictly than in the parent class (strengthening preconditions).

Follow-up & Advanced Questions
Follow-ups:
  • How does LSP relate to Interface Segregation Principle?
  • How do you fix code that violates LSP without removing inheritance?
Advanced:
  • Explain LSP in terms of design-by-contract (preconditions, postconditions, and invariants).
  • How does the compiler handle variance (out/in keywords) to enforce type safety in C# generic interfaces?
Interview Advice

Tips: Mention the phrase 'Design by Contract'. A child class can weaken preconditions (accept more inputs) or strengthen postconditions (return more specific outputs), but never the opposite.

Memory Trick: If it looks like a duck, quacks like a duck, but needs batteries to work, you probably have a violation of LSP.

Company Expectations: The interviewer wants to see if you can identify LSP violations in code reviews and refactor deep inheritance trees into clean interfaces or compositions.

Interface vs Abstract Class ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

An interface is a pure contract that defines method signatures (and default implementations in C# 8+) but cannot hold instance state. An abstract class is a partially implemented class that can define state, constructors, and default implementations, and supports single inheritance only.

30-Second Elevator Pitch
Perfect Answer

An interface defines a contract (can-do features) and support multiple inheritance. An abstract class defines a blueprint (is-a identity), supports code reusability with fields/constructors, but is limited to single inheritance. Use interfaces for decoupling, abstract classes for sharing implementation.

Deep-Dive Answer (2 Minutes)

The choice comes down to inheritance and state. A class can implement multiple interfaces, but can only inherit one abstract class. Abstract classes can have instance fields, non-public access modifiers (like protected), and constructors to initialize state. Interfaces, traditionally, only had public signatures. However, in modern C# (C# 8+), interfaces support static fields, private methods, and Default Interface Methods (DIM). Despite these features, interfaces still cannot hold instance state (non-static fields). Abstract classes represent an identity ('A SavingsAccount is an Account'), while interfaces represent a capability ('A class can be serialized via ISerializable').

Real-Time Metaphor

A smartphone. The brand (Apple, Samsung) is its abstract base (defines core architecture, battery connection, assembly logic). The capabilities (Wireless Charging, Bluetooth, Waterproofing) are interfaces. A phone can have multiple capabilities but has only one core brand architecture.

Real Project Implementation

In our order processing pipeline, we use an abstract class `BaseOrderProcessor` to manage state changes, database logging, and transaction blocks. However, for payment checkouts, orders can be paid via Stripe, PayPal, or Crypto. We define `IPaymentHandler` as an interface. The `StripeOrderProcessor` inherits from `BaseOrderProcessor` (single inheritance) and implements `IPaymentHandler`.

Visual Workflow
Abstract Class (Is-A)
Single Inheritance, holds instance state, has constructors, protected access
Interface (Can-Do)
Multiple Implementation, no instance state, defines generic capabilities
Internal Mechanics & Working

In the CLR, resolving an interface call (Interface Method Table - IMT lookup) is slightly slower than calling a virtual method on an abstract class (Virtual Method Table - vtable lookup), because multiple classes can implement the same interface, requiring dynamic resolution. However, modern JIT compilers optimize this difference to near zero.

Advantages & Disadvantages
Advantages

Abstract class: Great for framework design, easy to add methods without breaking children (if concrete). Interface: Highly decoupled, enables unit testing via mocking frameworks (Moq), supports multiple inheritance.

Disadvantages

Abstract class: Restricts child classes due to C#'s single inheritance model. Interface: Prior to default implementations, adding a method to an interface broke all implementing classes.

Best Practices
Architect Tips

1. Default to Interfaces for API design to ensure testability and mockability. 2. Use Abstract Classes when designing internal class hierarchies to prevent code duplication. 3. Make abstract class constructors protected so they cannot be instantiated directly.

Code Example
C# Source 
public interface IPayable
{
    void Pay(decimal amount); // Contract
}

public abstract class BaseEmployee
{
    public string Name { get; set; } // Instance State
    protected BaseEmployee(string name) => Name = name; // Constructor
    public abstract void Work();
}
Common Developer Pitfalls
Mistake Alert

Declaring fields in an interface. Interfaces cannot contain instance fields. Another mistake is using abstract classes for everything, which leads to rigid inheritance hierarchies that are hard to refactor.

Follow-up & Advanced Questions
Follow-ups:
  • Can an interface have concrete methods in C# 8+? Yes, using Default Interface Methods.
  • When would you use both together?
Advanced:
  • How do default interface methods handle diamond inheritance conflicts in C#?
  • Explain how IMT (Interface Method Table) works in CLR runtime compared to standard Virtual Method Table.
Interview Advice

Tips: If asked this, highlight that the lack of instance fields in interfaces is the main remaining architectural difference, even with the addition of default methods in C# 8.

Memory Trick: Abstract class = Shared genes (you can only have one set of biological parents). Interface = Skills (you can learn how to swim, code, and drive all at once).

Lazy Loading ⭐⭐⭐⭐⭐

Definition

30-Second Elevator Pitch
Perfect Answer

Deep-Dive Answer (2 Minutes)

To enable this in EF Core, you must install the `Microsoft.EntityFrameworkCore.Proxies` package, call `UseLazyLoadingProxies()` in DbContext setup, and declare navigation properties as `public virtual`. While it reduces initial payload size, it is a primary source of the N+1 query problem.

Scrolling through a social media feed. The app only loads the post text. The comments are not loaded until you click the 'View Comments' button, saving data and server resources.

We fixed this by switching to Eager Loading with `.Include()`.

Definition

Perfect Answer

AsNoTracking is used for read-only queries in EF Core. It tells EF Core not to cache the queried objects in the Change Tracker. This reduces memory usage, bypasses tracking overhead, and speeds up query execution. You cannot save updates back to the database for these entities.

Definition

Eager Loading is a query pattern in Entity Framework Core where related data is loaded from the database as part of the initial query using the '.Include()' and '.ThenInclude()' methods.

30-Second Elevator Pitch
Perfect Answer

Eager loading retrieves the main entity and its related entities in a single database roundtrip. In EF Core, this is done using the `.Include()` method. It prevents the N+1 query problem by loading all necessary data upfront, though it can result in complex SQL queries.

Deep-Dive Answer (2 Minutes)

Eager loading is the opposite of lazy loading. When you query an entity, you specify exactly which navigation properties should be fetched immediately. EF Core generates a SQL query with JOIN statements to pull all requested tables in a single command. In C#, we use `.Include(x => x.Relation)`. For nested relations, we chain `.ThenInclude()`. While this guarantees all data is loaded in one database roundtrip (preventing N+1 queries), developers must be careful not to load too many levels, as it can cause a Cartesian product explosion, making the database transfer massive amounts of duplicate data.

Real-Time Metaphor

Ordering a combo meal. You get the burger, fries, and drink all in one single tray (one trip). You don't have to go back to the counter three times to fetch each item separately.

Real Project Implementation

In our checkout API, we needed to validate an `Order` along with its `Items` and the `Product` details of each item. We wrote: `_context.Orders.Include(o => o.OrderItems).ThenInclude(oi => oi.Product).FirstOrDefault(o => o.Id == orderId)`. This loaded all three database layers in a single query, making validation fast and database interaction efficient.

When EF Core parses a LINQ expression with `.Include()`, it compiles the query into a SQL command that uses LEFT JOINs to fetch the related columns. It then runs identity resolution on the reader's output, stitching together the parent and child objects in memory.

Advantages & Disadvantages
Advantages

1. Executes only a single database roundtrip, which is highly efficient over network connections. 2. All related data is loaded and ready, avoiding any risk of ObjectDisposedException. 3. Predictable performance profile.

Disadvantages

1. Can cause a Cartesian Product issue when joining multiple collection navigation properties, returning too much redundant data. 2. SQL queries can become extremely complex and slow down database execution plans.

Best Practices
Architect Tips

Method Overloading vs Method Overriding ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

Method Overloading is defining multiple methods in the same class with the same name but different signatures (static polymorphism). Method Overriding is redefining a base class virtual or abstract method in a derived class with the same signature (dynamic polymorphism).

30-Second Elevator Pitch
Perfect Answer

Overloading is compile-time polymorphism where methods in the same class share a name but have different parameters. Overriding is runtime polymorphism where a child class changes the implementation of a base class virtual/abstract method using the `override` keyword.

Deep-Dive Answer (2 Minutes)

Method Overloading and Method Overriding are the two forms of polymorphism in C#. Overloading (Static or Compile-time Polymorphism) allows us to write multiple methods with the same name within the same class, but they must differ in their parameters (number, type, or order). The compiler decides which method to call at compile time based on the arguments passed. Overriding (Dynamic or Runtime Polymorphism) allows a derived class to provide a specific implementation of a method that is already defined in its base class. The base method must be marked `virtual`, `abstract`, or `override`, and the child method uses the `override` keyword. The CLR resolves which method to execute at runtime based on the actual type of the object, not the reference type.

Real-Time Metaphor

Overloading: A printer's 'Print()' method. You can call `Print(string text)`, `Print(Image img)`, or `Print(Document doc, int pages)`. Overriding: A generic 'Animal' making a sound. The base class has a virtual method `MakeSound()`. Derived classes like `Dog` override it to bark, and `Cat` override it to meow.

Real Project Implementation

In our Logging service, we overloaded `Log(string message)` and `Log(Exception ex, string message)` to handle different logging contexts in the same class. In our payment flows, our abstract `BasePaymentProcessor` has a virtual method `ValidateTransaction()`. The `PaypalProcessor` overrides this method to add specific validation checks for PayPal tokens.

Internal Mechanics & Working

For overloading, the compiler generates distinct method names in MSIL based on name mangling (signature checks). For overriding, the compiler marks the base method as `virtual` and the child method as `virtual final` or `override`. The CLR uses the object's header to locate its Virtual Method Table (vtable) at runtime, resolving the call to the most derived subclass implementation.

Advantages

Disadvantages

Overloading: Overuse with too many implicit type conversions can cause compiler ambiguity errors. Overriding: Tight coupling between base and child classes; changing base signatures requires updating all overrides.

ACTUALLY ASKED IN THIS COMPANY
Definition

30-Second Elevator Pitch
Perfect Answer

Dependency Injection is a technique where an object receives other objects that it depends on. Deep-Dive Answer (2 Minutes)

DI is a key pattern for achieving loose coupling. Instead of a class managing its own dependencies, the control is inverted: the framework's IoC container handles the instantiation and lifetime of dependencies and injects them at runtime. Services are typically requested by declaring them in a class constructor, and the IoC container automatically resolves them.

A restaurant. The chef (class) needs knives, pans, and ingredients (dependencies) to cook. Instead of the chef going out to buy them, the restaurant owner (DI container) provides them to the chef in the kitchen (constructor injection).

Async & Await ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

30-Second Elevator Pitch
Perfect Answer

Async and Await enable asynchronous programming. 'async' marks a method as asynchronous, and 'await' suspends execution without blocking the thread. The thread returns to the thread pool to handle other work while the task completes.

Deep-Dive Answer (2 Minutes)

Async/Await is used for non-blocking asynchronous operations, especially I/O tasks like database queries or API calls. When a method hits an `await` keyword, the compiler splits the method. The calling thread is released back to the Thread Pool to process other incoming requests. Behind the scenes, the compiler builds a state machine. Once the awaited I/O task completes, the operating system notifies the CLR. The state machine requests a thread from the Thread Pool to resume the rest of the method from where it left off. This prevents thread starvation and allows a small pool of threads to handle thousands of concurrent requests.

Real-Time Metaphor

Ordering coffee at a cafe. You place your order, get a buzzer (Task), and step aside (await). The cashier can take orders from other customers (thread not blocked). When your buzzer rings (Task complete), you collect your coffee and drink it (method resumes).

Real Project Implementation

In our API, we fetch order history from the database: `public async Task GetOrders() { var data = await _context.Orders.ToListAsync(); return Ok(data); }`. By using `ToListAsync()` and `await`, we release the IIS/Kestrel worker thread while SQL Server fetches the data, letting the API handle other incoming user requests.

Starts executing async method
Await I/O Task
Schedules task. Thread A returns to thread pool immediately.
I/O Execution
Operating system handles I/O. No active CPU thread is blocked.
Task Complete

Resumes executing the rest of the method
Internal Mechanics & Working

The compiler transforms an `async` method into a struct implementing `IAsyncStateMachine`. When `await` is called, it checks if the task is already completed. If not, it registers a callback (`MoveNext`) on the task's awaiter. The method returns a Task to the caller, and the thread yields. When the task completes, `MoveNext()` is executed on a thread pool thread, restoring the method's variables and executing the remaining code.

Advantages & Disadvantages
Advantages

1. Improves application throughput and scalability by releasing worker threads during I/O operations. 2. Keeps UI applications (like WPF or WinForms) responsive by offloading heavy tasks from the UI thread. 3. Simple, readable syntax that looks like synchronous code.

Disadvantages

1. State machine creation adds a small memory overhead. 2. Debugging can be more difficult because stack traces are split across threads.

1. Use 'Async all the way': avoid mixing synchronous and asynchronous code (e.g., using `.Result` or `.Wait()`), which can cause deadlocks. 2. Append 'Async' to the names of asynchronous methods. 3. Use `ConfigureAwait(false)` in library code to avoid capturing the synchronization context.

IEnumerable vs IQueryable ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

IEnumerable is an in-memory collection reader that evaluates queries client-side using LINQ to Objects. IQueryable is a database query provider that translates LINQ expressions into SQL and executes them server-side.

30-Second Elevator Pitch
Perfect Answer

IEnumerable is for in-memory collections and evaluates queries client-side. IQueryable is for database queries (like EF Core) and translates LINQ expressions into SQL, running them database-side. Always use IQueryable for database queries to filter data before pulling it into memory.

Deep-Dive Answer (2 Minutes)

Both interfaces are used to represent collections, but they differ in how and where queries are executed. `IEnumerable` is defined in `System.Collections`. It works with in-memory collections (like List or Array) using LINQ-to-Objects. When you filter an `IEnumerable`, it pulls all records into memory first, then filters them client-side in the application. `IQueryable` is defined in `System.Linq` and inherits `IEnumerable`. It is designed for out-of-memory data sources, like databases. It stores the query as an Expression Tree. When you apply filters (like `.Where()`), it appends them to the expression tree. The query is only executed when you iterate or call a termination method (like `.ToList()`). At that point, the query provider translates the expression tree into SQL (e.g., SELECT ... WHERE ...), executes it on the database server, and returns only the filtered records, saving network bandwidth and memory.

Real-Time Metaphor

Ordering books. IEnumerable is buying the entire bookstore inventory, bringing it home, and throwing away the books you don't want. IQueryable is looking at the catalog online, ordering the specific book you want, and having only that book delivered.

Real Project Implementation

This pulled all 50,000 user rows from SQL Server into the web server's memory, then filtered them. We changed it to `_context.Users.AsQueryable().Where(u => u.Country == "India")`. This executed a SQL `WHERE` clause on the database and returned only the 200 matching rows, improving page load times.

IEnumerable uses a delegate-based model (e.g., `Func`) and iterates using `IEnumerator`. IQueryable uses `Expression>`, representing code as data (an expression tree). The QueryProvider (like Entity Framework Provider) parses this expression tree, translates it into the target query language (SQL), and executes it.

Advantages

Disadvantages

Reference Type vs Value Type ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

Value Types in C# store their actual data directly and are typically allocated on the Stack. Reference Types store a reference (pointer) to the memory address of the data, which is allocated on the Heap.

30-Second Elevator Pitch
Perfect Answer

Value types (like int, struct, bool) store their data directly and are typically allocated on the stack. Reference types (like class, string, object) store a pointer to their data, which is allocated on the heap. Value types are copied by value, while reference types are copied by reference.

Deep-Dive Answer (2 Minutes)

C# types are divided into Value Types and Reference Types. Value Types include primitives (int, double), enums, and structs. They inherit from `System.ValueType`. When assigned to another variable, the actual data is copied. They are typically allocated on the Stack (unless they are part of a class instance allocated on the heap). Reference Types include classes, interfaces, delegates, and strings. They inherit from `System.Object`. When assigned, only the memory pointer is copied, meaning both variables point to the same object on the heap. Value types cannot be null (unless declared as Nullable), whereas reference types default to null. Value types are cleaned up when they go out of scope, while reference types are managed by the Garbage Collector.

Real-Time Metaphor

Value Type: Printing a physical document. You give a copy to a coworker. If they write on their copy, your document doesn't change. Reference Type: Sharing a Google Doc link. You give the link to a coworker. If they edit the document, you see the changes immediately because you are both accessing the same file.

Real Project Implementation

In our banking ledger, we represent money values using a custom `decimal` or `struct` (Value Type) to avoid references and ensure updates do not overwrite other accounts. We represent `Account` as a class (Reference Type) so that all services pointing to that account update the same shared instance.

Visual Workflow

Stack Pointer (Reference Type)

Heap Storage
Address 0x80F4 -> Stores actual object { Name: 'John', Age: 25 }
Internal Mechanics & Working

When a value type is declared, the JIT compiler allocates space directly in the active stack frame. When a reference type is declared, the JIT compiler allocates 4 or 8 bytes on the stack for the pointer. It then calls the `newobj` instruction, which requests the CLR GC heap to allocate memory for the object, initializes its sync block and type object pointer, and returns the heap address to the stack pointer.

Advantages & Disadvantages
Advantages

Value Types: Fast allocation and cleanup on the stack, no garbage collection overhead. Reference Types: Efficient passing of large objects because only the pointer is copied, not the entire object.

Disadvantages

Value Types: Passing large structs by value can cause performance issues due to data copying. Reference Types: Heap allocation overhead and garbage collection pauses.

Best Practices
Architect Tips

1. Use classes for most business logic objects. 2. Keep structs small (typically under 16 bytes) and immutable to avoid copying overhead. 3. Avoid boxing and unboxing (converting value types to reference types and back), which adds heap allocation overhead.

Heap Memory vs Stack Memory ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

Stack Memory is a fast, structured memory block allocated per thread that stores local variables and method execution frames. Heap Memory is a large, unstructured memory pool shared across all threads used for dynamic object allocations, managed by the Garbage Collector.

30-Second Elevator Pitch
Perfect Answer

Stack is fast, self-managing memory that stores local variables and method call frames. Heap is larger, slower memory used for dynamic allocations (objects). The stack runs in a LIFO order, while the heap is managed by the Garbage Collector.

Deep-Dive Answer (2 Minutes)

Stack and Heap are the two memory areas managed by the CLR. The Stack is allocated per thread. It operates in a Last-In, First-Out (LIFO) order. When a method is called, a stack frame is pushed. It stores local variables and method parameters. When the method exits, the stack frame is popped, and memory is reclaimed instantly. This makes stack allocation incredibly fast. The Heap is a shared pool of memory used for dynamic allocations. When you instantiate a class with the `new` keyword, the object is created on the Heap, and its memory address is stored in a stack pointer. Heap allocation requires finding free space and is managed by the Garbage Collector, which periodically runs to reclaim memory from unused objects, adding some performance overhead.

Real-Time Metaphor

Stack: A stack of dinner plates. You add plates on top and remove them from the top. It is organized and fast to manage. Heap: A large warehouse floor. Items are placed wherever there is open space. Finding and cleaning up items takes time and requires a supervisor (Garbage Collector).

Real Project Implementation

In our payment processing loop, we avoided creating new objects on the heap. Instead of instantiating new helper classes inside a loop that processed 10,000 transactions, we used structs and stack-allocated spans (`Span`). This reduced heap allocation, lowered Garbage Collector overhead, and prevented CPU spikes.

Visual Workflow
Stack Memory (Fast, Thread-Local)
Method Frame: stores int x, bool flag, and reference pointers. Cleaned up instantly on method exit.
Heap Memory (Large, Shared)
Stores Class Instances, Arrays, Strings. Cleaned up by Garbage Collector.
Internal Mechanics & Working

The stack pointer register in the CPU tracks the top of the stack. Stack allocations simply increment or decrement this register, taking 1 CPU cycle. The Heap uses a managed heap allocator. It maintains a pointer to the next available block of space (NextObjPtr). Allocating on the heap involves updating this pointer and checking if a garbage collection run is needed. The GC uses generations (Gen 0, 1, 2) to optimize collection times.

Advantages & Disadvantages
Advantages

Stack: Extremely fast, self-cleaning, thread-safe, no fragmentation. Heap: Allows objects to live beyond the scope of a single method, supports large data allocations.

Disadvantages

Instead, we use a global exception handling middleware to catch unhandled errors and return a standardized response.

ACTUALLY ASKED IN THIS COMPANY
Definition

A JSON Web Token (JWT) is an open standard (RFC 7519) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object, signed using a secret key.

30-Second Elevator Pitch
Perfect Answer

Deep-Dive Answer (2 Minutes)

JSON Web Token (JWT) is commonly used for authentication in modern web APIs. When a user logs in successfully, the server generates a JWT signed with a secret key (using algorithms like HMAC SHA256) and returns it to the client. The client stores it (usually in localStorage or HTTP-only cookies) and sends it in the `Authorization: Bearer ` header of subsequent requests. A JWT has three parts separated by dots: 1. Header: Specifies the token type and signing algorithm. 2. Payload: Contains claims (user data like UserID, Username, Expiry). 3. Signature: Created by hash-combining the base64-encoded header, payload, and a server-side secret key. Because the token is signed, the server can verify its integrity without querying a database.

Real-Time Metaphor

A concert ticket. The ticket has a header (ticket type), details (seat number, event date), and a barcode (signature). The venue staff scan the barcode to verify the ticket is authentic and hasn't been modified, without needing to look up your booking in the database.

In our Angular and ASP.NET Core system, users log in at `/api/auth/login`. On success, the API generates a JWT containing claims for `NameIdentifier` and `Email` with a 1-hour expiration. The Angular application interceptor automatically grabs this token from storage and appends it to all subsequent API requests.

Internal Mechanics & Working

When an API receives a request, the JWT middleware extracts the token from the Bearer header. It splits the token into its header, payload, and signature parts. Advantages

ACTUALLY ASKED IN THIS COMPANY
Definition

30-Second Elevator Pitch
Perfect Answer

A BehaviorSubject is a type of Subject in RxJS that requires an initial value and stores the current value. When a new subscriber joins, they immediately receive the latest value, and they also receive any subsequent updates.

Deep-Dive Answer (2 Minutes)

In Angular, we use RxJS Subjects for state management. A standard `Subject` does not store a value and only emits updates that occur after the subscription starts. A `BehaviorSubject` is different: 1. It requires an initial value (e.g., `new BehaviorSubject(false)`). 2. It stores the current value. You can access this value synchronously at any time using the `.value` property or `.getValue()`. 3. It is stateful: when a new component subscribes, the BehaviorSubject immediately sends its current value, even if no new update has occurred. This makes it ideal for sharing global state, like user login status or shopping cart items, across components.

Real-Time Metaphor

A cassette player vs a podcast. A standard Subject is like joining a radio show (you only hear what is playing now; you missed the start). A BehaviorSubject is like a podcast (it starts playing the latest recorded episode as soon as you subscribe).

Real Project Implementation

We built a `CartService` in our store application. It maintains a `BehaviorSubject(0)` tracking the number of items in the cart. The header cart counter and the checkout page both subscribe to this subject. When a user adds an item, the counter updates immediately across all pages.

Internal Mechanics & Working

BehaviorSubject inherits from Subject. Internally, it overrides the `_subscribe` method. When a new observer subscribes, it first validates the subscription and then calls the observer's `next()` method, passing its internal `_value` variable. Subsequent calls to `.next(val)` update `_value` and loop through all registered observers to deliver the new value.

Advantages

1. Always provides the current value to new subscribers, avoiding timing and race condition bugs. 2. Allows synchronous value retrieval using `.value` without needing to subscribe. 3. Perfect for sharing global state across decoupled components.

Stored Procedure vs Function ⭐⭐⭐⭐⭐
ACTUALLY ASKED IN THIS COMPANY
Definition

A Stored Procedure is a precompiled set of SQL statements that can perform transactions and database modifications. A User-Defined Function (UDF) is a database routine designed to calculate and return a value or table.

30-Second Elevator Pitch
Perfect Answer

Stored Procedures are designed to perform database modifications and transactions, and can return multiple datasets. Functions are designed to perform calculations and must return a single value or table, cannot modify the database, and can be used directly in SQL SELECT queries.

Deep-Dive Answer (2 Minutes)

Stored Procedures and User-Defined Functions are database routines, but they have different purposes: 1. Stored Procedures: Called using the `EXEC` statement. They can perform data modification operations (INSERT, UPDATE, DELETE), manage transactions (COMMIT, ROLLBACK), and return multiple datasets. They support both input and output parameters. 2. Functions: Must return a value (scalar or table). They cannot perform data modification operations (read-only) and cannot manage transactions. Because they are deterministic and return values, they can be called directly inside SQL `SELECT`, `WHERE`, or `JOIN` statements (e.g., `SELECT dbo.CalculateTax(Price) FROM Products`). Stored procedures cannot be used inside SELECT queries.

Real-Time Metaphor

Stored Procedure: A factory manager. They can hire workers, build parts, move inventory, and give you a status report (modifies database, runs transactions). Function: A calculator. You type in numbers, and it returns a result without changing anything in the office (read-only calculation).

Real Project Implementation

In our invoicing database, we use a Function `dbo.CalculateDiscount(CustomerId, TotalAmount)` to calculate invoice discounts inline in reports: `SELECT Id, Total, dbo.CalculateDiscount(CustId, Total) FROM Invoices`. For generating the end-of-month financial statements, we call a Stored Procedure `sp_GenerateLedgerEntry` because it needs to insert auditing rows and run a transaction block.

Internal Mechanics & Working

SQL Server compiles Stored Procedures and caches their execution plans. Functions are also compiled, but Inline Table-Valued Functions (TVFs) are expanded into the calling query like a view, allowing the SQL Query Optimizer to evaluate indexes across both. Multi-Statement Table-Valued Functions (MSTVFs) create temporary table variables internally, which can slow down performance because they lack column statistics.

Advantages & Disadvantages
Advantages

Stored Procedure: Supports full transactions, reduces network traffic for complex processes, hides database schema. Function: Can be used inline in SELECT statements, highly reusable for calculations, table-valued functions behave like parameterized views.

Disadvantages

Stored Procedure: Hard to unit-test and trace in version control, consumes database server CPU. Function: Multi-statement functions can slow down performance because they lack statistics and cannot run parallel queries.

Best Practices
Architect Tips

1. Use Stored Procedures for operations that modify data or run transactions. 2. Use Inline Table-Valued Functions instead of Multi-Statement Functions to help the query optimizer build better execution plans. 3. Avoid running complex business logic inside database functions; perform calculations in the application layer if possible.

Q: What are the different types of Joins in SQL Server?

Joins are used to combine rows from two or more tables based on a related column. The types are: 1. INNER JOIN: Returns matching rows in both tables. 2. LEFT JOIN (Outer): Returns all rows from the left table, and matched rows from the right table. If no match, NULL is returned. 3. RIGHT JOIN (Outer): Returns all rows from the right, and matched from the left. 4. FULL JOIN (Outer): Returns all rows when there is a match in either left or right table. 5. CROSS JOIN: Returns the Cartesian product (all combinations) of both tables.

Q: Explain Clustered vs Non-Clustered Indexes.

Indexes speed up data retrieval. - Clustered Index: Sorts and stores the physical data rows in the table based on the key columns. A table can have only ONE Clustered Index (usually the Primary Key). - Non-Clustered Index: Creates a separate structure containing the key values and pointers (row locators) to the actual data rows. You can create multiple Non-Clustered Indexes on a table to speed up different queries.

Q: What is a CTE and when should you use it?

A Common Table Expression (CTE) is a temporary result set defined within the execution scope of a single SELECT, INSERT, UPDATE, or DELETE statement. It is declared using the `WITH` keyword. It improves query readability, is a cleaner alternative to nested subqueries, and supports recursion (Recursive CTEs) to query hierarchical data like reporting structures or folder systems.

Q: Difference between Temp Table and Table Variable.

- Temp Tables (#TempTable): Stored in the tempdb database, support indexes, transaction logs, and statistics. They can be shared across nested queries and are best for large datasets (>10,000 rows). - Table Variables (@TableVar): Stored in memory (spill to tempdb if large), have no statistics (can cause poor query plans for large data), do not participate in transactions, and are best for small lookup sets.

Q: Explain database Normalization up to 3NF.

Normalization organizes data to reduce redundancy and improve integrity: 1. 1NF (First Normal Form): Eliminate duplicate columns, ensure cell values are atomic, and define a primary key. 2. 2NF: Meet 1NF and remove partial dependencies (every non-key column must depend on the entire primary key, not just part of a composite key). 3. 3NF: Meet 2NF and remove transitive dependencies (non-key columns must not depend on other non-key columns).

Q: What is a Deadlock and how do you resolve it?

A Deadlock occurs when Session A holds a lock on Table 1 and requests a lock on Table 2, while Session B holds a lock on Table 2 and requests a lock on Table 1. Both sessions block indefinitely. SQL Server detects this, chooses a 'deadlock victim', rolls back their transaction, and throws error 1205. To resolve: query tables in the same order, use index optimization to shorten lock times, keep transactions small, and use the NOLOCK hint or Snapshot Isolation if appropriate.

Q: How do you optimize a slow running query?

1. Analyze the Execution Plan: Look for heavy operations like Table Scans or Index Scans, which indicate missing indexes. Look for Key Lookups. 2. Create Missing Indexes: Add non-clustered indexes, using `INCLUDE` to cover query columns. 3. Update Statistics: Run `UPDATE STATISTICS` to ensure the optimizer has accurate data distribution tables. 4. Avoid Wildcards: Avoid using leading wildcards like `LIKE '%search'` which prevent index usage (forces scan). 5. Use SARGable Queries: Avoid applying functions to index columns in WHERE clauses (e.g. use `CreatedDate >= @date` instead of `YEAR(CreatedDate) = 2026`).

Angular Core

Q: What is Angular?

Angular is a TypeScript-based open-source front-end web application framework maintained by Google for building single-page applications (SPAs) and dynamic web applications. In modern Angular (v14+), it has transitioned toward standalone components, reducing reliance on NgModules.

Q: What are the key features of Angular?

Key features include: 1. Component-based architecture for building reusable UI blocks. 2. TypeScript for static typing and early error checks. 3. Dependency Injection (DI) for loose coupling. 4. Declarative templates with data binding. 5. Built-in routing and HTTP client service. 6. Form handling systems (Reactive and Template-driven). 7. Modern performance features: Signals (v16+), standalone components, and built-in SSR support (v17+).

Q: Explain the difference between AngularJS and Angular.

AngularJS (Angular 1.x) is a JavaScript framework based on MVC (Model-View-Controller) architecture using controllers and scope ($scope). Angular (Angular 2+ up to the latest v21) is a complete rewrite based on TypeScript and a Component-based architecture. Angular offers much faster performance, mobile-first optimization, modular structure, better CLI support, and advanced features like change detection strategies and Signals.

Q: What is TypeScript and why is it used in Angular?

TypeScript is a strongly typed superset of JavaScript that compiles to plain JavaScript. It is used in Angular because it offers static typing, interfaces, advanced object-oriented features (like classes and decorators), and excellent IDE autocompletion and compile-time error checks. This makes large-scale application development much more stable and maintainable.

Q: What is data binding in Angular?

Data binding is the synchronization of data between the component class (model) and the HTML template (view). Angular supports: 1. One-way Data Binding: Component to View (Interpolation `{{}}`, Property binding `[]`), or View to Component (Event binding `()`). 2. Two-way Data Binding: Bi-directional synchronization using the `[(ngModel)]` directive.

Q: What is interpolation in Angular?

Interpolation is a way to display component properties or evaluate expressions inside HTML templates. It is denoted by double curly braces `{{ expression }}`. At runtime, Angular evaluates the expression and inserts the string result into the DOM.

Q: What is property binding in Angular?

Property binding allows you to bind a component property value to an HTML element property or directive attribute, using square brackets. Example: `` binds the `src` attribute of the image element to the `imageUrl` field in the component class.

Q: Differentiate between one-way and two-way data binding in Angular.

- One-way Data Binding: Data flows in a single direction. It either flows from the component class to the template (using `{{}}` or `[property]`) or from the template to the component class via event triggers (using `(event)='handler()`). - Two-way Data Binding: Data flows in both directions simultaneously. Any change in the UI input updates the component model, and any program update to the model immediately updates the UI input. It is implemented using `[(ngModel)]`.

Q: What is Angular CLI? How do you use it?

Angular CLI (Command Line Interface) is a command-line tool used to initialize, develop, build, scaffold, and maintain Angular applications. Common commands include: - `ng new ` (creates a new project) - `ng serve` (runs the development server) - `ng generate component/service/guard ` (scaffolds new files) - `ng build` (compiles the app for production)

Q: What is a component in Angular?

A component is the fundamental building block of an Angular UI. It consists of: 1. A TypeScript class: Handles data, logic, and methods. 2. An HTML Template: Defines the layout structure. 3. CSS Styles: Defines styling (scoped to the component by default). It is declared using the `@Component` decorator, which specifies selectors, templates, and styles.

Q: How do you create a component in Angular?

You generate a component using the Angular CLI command: `ng generate component ` (or `ng g c `). This automatically creates the TypeScript, HTML, CSS, and spec test files, and adds the component to the imports of its parent standalone component or NgModule.

Q: Explain the role of NgModule in Angular.

An NgModule (declared via `@NgModule`) is a container used to group related components, directives, pipes, and services into cohesive blocks of functionality. It declares which components belong to the module, imports dependency modules, and exports components for public use. In modern Angular (v15+), NgModules are optional and replaced by standalone components.

Q: What is Angular routing? How do you implement routing in Angular?

Angular routing is a mechanism that allows navigation between different component views in a Single Page Application (SPA) based on the browser URL path. It is implemented by configuring a `Routes` array, defining paths linked to components, providing the routing configurations via `provideRouter` (in modern apps) or `RouterModule.forRoot()`, and placing a `` tag in the main layout HTML.

Q: What are Angular directives? Provide examples.

Directives are instructions in the template DOM that tell Angular how to alter the appearance or behavior of HTML elements. Examples include: 1. Structural Directives: Alter DOM layout structure by adding/removing elements (e.g. `*ngIf`, `*ngFor`). 2. Attribute Directives: Alter the appearance or behavior of an element without changing the DOM structure (e.g. `ngStyle`, `ngClass`).

Q: Explain the difference between structural directives and attribute directives in Angular.

- Structural Directives: Modify the DOM layout. They add, remove, or manipulate DOM elements. They are prefixed with an asterisk (*), such as `*ngIf` and `*ngFor` (or using modern Angular `@if` and `@for` control flow syntax). - Attribute Directives: Modify the properties or behavior of an existing DOM element, component, or other directive. They do not change DOM structure. Examples include `[ngClass]`, `[ngStyle]`, or custom directives that listen to hover events.

Q: What is dependency injection in Angular? Why is it used?

Dependency Injection (DI) is a design pattern where a class receives its dependencies from an external source rather than instantiating them itself. Angular has a hierarchical DI framework. It is used to inject services (like HttpClient or custom services) into components, promoting decoupling, code reusability, and simple mocking during unit testing.

Q: How do you handle forms in Angular?

Angular provides two workflows to build and manage forms: 1. Template-driven Forms: Best for simple forms. Validation rules and bindings are configured inside the HTML template using directives like `ngModel`. 2. Reactive Forms: Best for complex forms. The form structure and validation rules are configured programmatically in the TypeScript class using `FormGroup`, `FormControl`, and `FormBuilder`, offering better control, testability, and RxJS integration.

Q: What are template-driven forms in Angular?

Template-driven forms rely on HTML templates to build and manage the form model. You bind inputs using `ngModel` and manage validations using standard HTML attributes (like `required` or `pattern`). Angular automatically builds the underlying FormModel in the background. They are simple to write but harder to unit-test and validate dynamically.

Q: What are reactive forms in Angular?

Reactive Forms provide a programmatic, model-driven approach to form management. The form structure is defined in the TypeScript component class using `FormGroup`, `FormControl`, and `FormArray`. This provides direct access to form validation states and value change streams via RxJS observables, making them highly testable and flexible for dynamic forms.

Q: Explain the difference between template-driven forms and reactive forms.

- Model Location: Template-driven forms define the form model in the HTML template. Reactive forms define the model programmatically in the TypeScript class. - Data Flow: Template-driven forms use asynchronous data flows (two-way binding). Reactive forms use synchronous, observable-driven data flows. - Validation: Template-driven forms use directives in HTML. Reactive forms use validator functions in TypeScript. - Testing: Reactive forms are much easier to unit-test because you can test validation logic without rendering the DOM.

Q: How do you perform validation in Angular forms?

Angular provides built-in validator functions (like `Validators.required`, `Validators.minLength`, `Validators.pattern`). In Reactive Forms, these are passed to the `FormControl` constructor. In Template-driven forms, they are added as HTML attributes. You can also create Custom Validators by writing functions that implement the `ValidatorFn` interface, returning `null` if valid, or an error object if invalid.

Q: What are pipes in Angular? Provide examples.

Pipes are template utilities used to transform raw data into a formatted string directly inside HTML views. Examples of built-in pipes include `date`, `currency`, `uppercase`, `lowercase`, and `json`. You can create custom pipes by implementing the `PipeTransform` interface and writing a custom `transform()` method.

Q: Explain the concept of Angular services.

An Angular service is a class designed to encapsulate reusable business logic, coordinate state sharing, or execute external API requests. Services are typically registered as singletons using the `@Injectable({ providedIn: 'root' })` decorator. This keeps components focused strictly on managing view logic and user interactions.

Q: How do you create a service in Angular?

You generate a service using the Angular CLI command: `ng generate service ` (or `ng g s `). This generates the service class and its testing spec file. You write your logic inside the class and inject it into component constructors.

Q: What is HTTPClient in Angular? How do you use it to make HTTP requests?

HttpClient is a built-in Angular service that performs HTTP communications. It is configured by providing the HttpClient provider (`provideHttpClient()` in modern Angular) and injecting the `HttpClient` service into your data services. It returns RxJS Observables, supporting typed responses and request methods like `.get()`, `.post()`, and `.delete()`.

Q: What are observables in Angular?

Observables are asynchronous data streams managed by the RxJS library. Unlike Promises which handle single events, Observables can emit multiple values over time. They are used in Angular for HTTP client requests, routing parameters, forms validation value updates, and custom event streams. They support powerful mapping and filtering operators (like `map`, `filter`, `switchMap`).

Q: Explain the concept of Angular modules.

Angular modules (NgModules) are containers that organize components, directives, pipes, and services into functional groups. Every Angular app historically had at least one root module (AppModule) to bootstrap the application. In modern Angular architectures, modules are largely replaced by standalone components, which import their dependencies directly.

Q: What is lazy loading in Angular? How do you implement it?

Lazy loading is a routing technique that delays the loading of feature modules or components until the user actually navigates to their routes. This reduces the initial bundle size and speeds up initial page load times. It is implemented in routes by using the `loadChildren` property with dynamic imports: `loadChildren: () => import('./feature/feature.routes').then(r => r.ROUTES)`.

Q: What is Angular material?

Angular Material is a UI component library built by Google for Angular developers. It implements Google's Material Design guidelines, offering a suite of pre-built, responsive, accessible, and easily stylable UI widgets like buttons, dialog modals, tables, input forms, and navigation sidebars.

Q: How do you handle authentication in Angular applications?

Authentication is managed via services and route guards: 1. Login: The client posts credentials to the API, receives a JWT, and stores it in LocalStorage. 2. Interception: An HTTP Interceptor automatically attaches the JWT bearer token to the Authorization header of all outgoing backend API calls. 3. Guards: Route Guards (like `CanActivate`) inspect the auth state before letting the user navigate to protected dashboards, redirecting unauthorized traffic to the login page.

Q: What are Angular guards? How do you use them?

Route Guards are security middleware that control routing access. You define guard functions (or services) implementing interfaces like `CanActivate` (blocks entering a route), `CanDeactivate` (blocks leaving a route), or `Resolve` (pre-fetches API data). You register them in your route definitions using properties like `canActivate: [AuthGuard]`.

Q: Explain the concept of change detection in Angular.

Change detection is the mechanism Angular uses to sync view templates with component data changes. Angular historically used `Zone.js` to intercept async events (clicks, timers, HTTP calls) and run a global check across the component tree. In modern Angular, performance can be optimized using `ChangeDetectionStrategy.OnPush` (only checks component when inputs change or events trigger) or native Signals.

Q: What is AOT compilation in Angular?

Ahead-of-Time (AOT) compilation compiles Angular HTML templates and TypeScript code into efficient JavaScript code during the build phase, before the browser runs the app. This is the default in modern Angular. It results in faster startup times, smaller bundle sizes (as the Angular compiler is excluded), and catches template errors at compile time.

Q: What is SSR (Server-Side Rendering) in Angular?

Server-Side Rendering (SSR) renders Angular components into static HTML pages on a Node.js server before sending them to the browser. In modern Angular (v17+), this is integrated via `@angular/ssr`. SSR improves initial page load times, makes the app feel faster, and enables search engine crawler indexing (SEO).

Q: How do you handle errors in Angular applications?

1. Local Handling: Wrapping async calls or API requests in try-catch blocks or using RxJS `catchError` operators. 2. Global Handling: Creating a custom class that implements the `ErrorHandler` interface. You override `handleError(error)` to intercept and log all unhandled application errors to logging services (like Application Insights), preventing the app from crashing silently.

Q: What are the different types of testing in Angular?

Angular supports: 1. Unit Testing: Testing individual components, services, or pipes in isolation using Jasmine and Karma. 2. Integration Testing: Testing components along with their rendered templates using the `TestBed` helper. 3. End-to-End (E2E) Testing: Testing the entire application flow in a real browser using modern tools like Playwright or Cypress.

Jasmine is a behavior-driven development (BDD) testing framework for JavaScript. It provides functions to structure test suites (`describe`), declare individual unit test specs (`it`), verify expectations (`expect().toBe()`), and create mock spy objects (`jasmine.createSpyObj()`) to test components without calling real APIs.

Karma is a test runner tool created by the Angular team. It spawns a web browser, runs the unit test suites written in Jasmine, monitors code changes, and displays test outcomes (success, fail, coverage report) directly in the command terminal.

Protractor was Angular's original, Selenium-based end-to-end testing framework. **Important Note:** Protractor has been deprecated by the Angular team and is removed from modern projects. Angular Universal was the original technology used to render Angular applications on the server side (SSR). **Important Note:** In Angular 17+, Angular Universal has been merged directly into the core Angular framework. Lifecycle hooks are callback methods Angular runs at specific stages of a component's lifecycle (creation, change checking, view rendering, destruction). Key hooks are `ngOnChanges`, `ngOnInit`, `ngDoCheck`, `ngAfterContentInit`, `ngAfterViewInit`, and `ngOnDestroy`.

Q: Explain the ngOnChanges lifecycle hook.

Q: What is the DbContext and how does the Change Tracker work?

The following questions are compiled from real-world candidate interviews. They cover C# & OOP, ASP.NET Core, and SQL & Entity Framework Core.

C# & OOP Real Interview Qs

Q: Why do we use the using keyword in C#?

The `using` keyword has two primary uses: 1) As a directive, to import namespaces. 2) As a statement, to define a scope where disposable objects are created. The using statement guarantees that `Dispose()` is called on objects implementing the `IDisposable` interface (like database connections or file streams) as soon as the scope exits, even if an exception occurs, preventing resource leaks.

Q: What is the difference between Class and Structure?

1) Type: Class is a Reference Type; Struct is a Value Type. 2) Memory Allocation: Classes are allocated on the Heap; Structs are typically allocated on the Stack. 3) Inheritance: Classes support full inheritance and can act as base classes; Structs do not support inheritance. 4) Nullability: Class variables can be null; Struct variables cannot be null (unless declared as Nullable, e.g. `int?`). 5) Copying: Assigning a class copies the pointer reference; assigning a struct copies the entire value state.

Q: Differentiate between Method Overloading and Method Overriding.

- Method Overloading: Static (compile-time) polymorphism. It defines multiple methods in the same class with the same name but different signatures (parameter types, count, or order). Resolved by the compiler. - Method Overriding: Dynamic (runtime) polymorphism. A derived class redefines a virtual or abstract method inherited from a base class using the `override` keyword. Resolved at runtime based on the actual object type via vtable.

Q: What is the difference between Abstract Class and Interface?

- Inheritance: A class can inherit only one abstract class but can implement multiple interfaces. - State: Abstract classes can hold instance state (non-static fields); interfaces cannot hold instance state. - Constructors: Abstract classes can define constructors; interfaces cannot. - Access Modifiers: Abstract classes support all access modifiers (protected, internal, private); interfaces traditionally only defined public contracts, although modern C# (C# 8+) supports private/static helper methods.

Q: Explain the Four Pillars of OOP.

1. Encapsulation: Grouping variables and methods into a single unit (class) and restricting direct access to protect data integrity (using access modifiers). 2. Abstraction: Hiding internal complexity and showing only essential interfaces to users. 3. Inheritance: Creating new classes that reuse, extend, and modify the behavior of existing classes. 4. Polymorphism: The ability of different classes to respond to the same method call in unique ways (static via overloading, dynamic via overriding).

Q: What are the SOLID Principles?

SOLID is an acronym for five object-oriented design principles: - S (Single Responsibility): A class should have only one reason to change. - O (Open/Closed): Classes should be open for extension but closed for modification. - L (Liskov Substitution): Derived classes must be substitutable for their base classes without breaking the program. - I (Interface Segregation): Clients should not be forced to implement interfaces they do not use. - D (Dependency Inversion): High-level modules should not depend on low-level modules; both should depend on abstractions.

Q: What is the difference between Value Type and Reference Type?

- Value Types: Directly store their data values. Allocated on the stack (unless inside a class). Examples: primitive types (int, bool), structs, enums. Copying copies the actual value. - Reference Types: Store a pointer to the memory address of the data allocated on the managed heap. Examples: classes, strings, interfaces, arrays. Copying copies only the pointer reference.

Q: Explain the difference between Equals() and == in C#.

- `==` operator: Compares reference identity for reference types (checks if pointers point to the same heap address) and compares values for value types by default. - `Equals()` method: A virtual method defined in `System.Object` that can be overridden by a class to perform structural value comparison (e.g. comparing string values rather than their heap addresses).

Q: Explain ref, out, and in parameters.

- `ref`: Passes an argument by reference. The variable must be initialized before passing, and the called method can read or modify it. - `out`: Passes an argument by reference. The variable does not need to be initialized before passing, but the called method is required to assign a value to it before returning. - `in`: Passes an argument by reference as read-only. The variable must be initialized before passing, and the called method cannot modify its value.

Q: What is the difference between const and readonly?

- `const`: Compile-time constants. Their values are hardcoded during compilation and cannot be changed. They can only be declared for primitive types and strings. - `readonly`: Runtime constants. Their values are evaluated at runtime and can be set either at declaration or inside the class constructor, allowing for dynamic initialization.

Q: What is Boxing and Unboxing?

- Boxing: The process of converting a value type (stack) to a reference type (heap) by wrapping it in a generic object instance. - Unboxing: The process of converting a reference type (boxed object on the heap) back to a value type (stack) by extracting its value. - Performance: Both operations have high CPU and memory overhead due to heap allocation and casting validation checks.

Q: What is a Delegate?

A delegate is a type-safe object representing reference pointers to methods. It defines a method signature (return type and parameters) and can hold references to any method matching that signature, enabling event handling, callbacks, and passing methods as arguments.

Q: What is a Partial Class?

A partial class allows the definition of a single class to be split across multiple `.cs` files. The compiler merges all partial definitions into a single compiled class at build time, which is useful when working with auto-generated code and manual code additions.

Q: Difference between Static Class and Singleton Pattern.

- Static Class: A language-level construct. It cannot be instantiated, cannot implement interfaces, cannot inherit from other classes, and all its members must be static. - Singleton Pattern: A design pattern. It enforces a single instance of a class, supports lazy initialization, can implement interfaces, supports inheritance, and manages its own lifetime.

Q: What is an Extension Method?

An extension method allows you to add new methods to existing classes or interfaces without modifying their source code. They are defined as static methods inside static classes, and their first parameter is prefixed with the `this` keyword to specify the type being extended.

Q: Managed vs Unmanaged Code in C#.

- Managed Code: Code executed by the CLR (Common Language Runtime) virtual machine. It benefits from automatic memory management (Garbage Collection), type safety validations, and runtime security controls. - Unmanaged Code: Code executed directly by the OS outside the CLR boundaries. Examples: C/C++ DLLs, Win32 APIs, where the developer must manage memory manually.

Q: What is Serialization and Deserialization?

- Serialization: The process of converting an object's state in memory into a storage-friendly stream format (like JSON, XML, or binary) to save or transmit it. - Deserialization: The reverse process, converting a stored format stream back into an active object instance in memory.

Q: Differentiate between Array and ArrayList.

- Array: Strongly-typed, fixed-size collection. It stores elements of a specific type, preventing runtime casting errors, and has fast indexing performance. - ArrayList: Weakly-typed, dynamically-sized collection. It stores elements as `System.Object`, meaning it can store mixed types, but requires boxing/unboxing, which is slower and less type-safe.

Q: What is the difference between .NET Framework and .NET Core?

- .NET Framework: The legacy development platform. Windows-only, relies on IIS for web apps, and is no longer under active development (version 4.8 is the last release). - .NET Core (now .NET 5/6/7/8/9): The modern development platform. Open-source, cross-platform (runs on Windows, Linux, macOS), optimized for microservices and cloud deployments, supports side-by-side versioning, and offers much faster performance.

Q: Monolithic Architecture vs Clean Architecture.

- Monolithic Architecture: The entire application, database layers, and UI are built and deployed as a single, combined unit. Simple to start but hard to scale and maintain as the codebase grows. - Clean Architecture (Onion/Hexagonal): The application is split into concentric layers (Domain, Application, Infrastructure, Presentation) with strict dependency rules pointing inward. This decouples business logic from frameworks and databases, enhancing testability and maintainability.

ASP.NET Core & Web API Real Interview Qs

Q: What is custom Middleware and how do you write it?

Middleware is a software component configured in Program.cs that intercepts and handles HTTP requests and responses in a pipeline. To write custom middleware, create a class with a constructor accepting `RequestDelegate` (the next middleware in the pipeline) and implement an async method named `Invoke` or `InvokeAsync` taking `HttpContext` to execute custom pre/post routing logic.

Q: What is the difference between Startup.cs and Program.cs in modern .NET?

In older versions (.NET Core 2.x/3.x), `Program.cs` bootstrapped the host and `Startup.cs` configured services (`ConfigureServices`) and the middleware pipeline (`Configure`). In modern .NET (6/8/9), `Startup.cs` is deprecated, and both service registration and pipeline configuration are consolidated directly into a single `Program.cs` file using the `WebApplicationBuilder` API.

Q: What are the types of Dependency Injection lifetimes in ASP.NET Core?

1. Transient (`AddTransient`): A new instance is created every time the dependency is requested. 2. Scoped (`AddScoped`): A single instance is created per HTTP request scope and shared across classes resolving it in that request. 3. Singleton (`AddSingleton`): A single instance is created once during startup and shared across the entire application lifetime.

Q: What is the difference between IActionResult and ActionResult?

- `IActionResult`: An interface representing the result of an action method (e.g. OK, NotFound, BadRequest). It allows returning any HTTP status response but doesn't specify the return type, making unit testing harder. - `ActionResult`: A helper class (introduced in ASP.NET Core 2.1) that allows returning either an implementation of `IActionResult` or a direct object of type `T`. It provides better type safety and cleaner unit testing.

Q: What is the purpose of HttpClientFactory?

HttpClientFactory is a built-in service used to create and manage `HttpClient` instances. In legacy systems, instantiating `HttpClient` manually using the `using` block causes socket exhaustion under load (sockets remain open in TIME_WAIT state). HttpClientFactory manages the underlying HTTP handler pool and reuse, preventing socket depletion while applying automatic logging and transient fault policies.

Q: What is CORS and how do you enable it in Web API?

CORS (Cross-Origin Resource Sharing) is a browser-enforced security feature that restricts web applications on one domain from making API calls to another domain. To enable it, register CORS policies in Program.cs using `builder.Services.AddCors(...)` and apply the middleware using `app.UseCors("PolicyName")` before routing and authorization.

Q: What are DTOs and why do we use them?

Data Transfer Objects (DTOs) are simple classes containing only data properties without business logic. They carry data between the Web API and client applications. We use them to avoid exposing internal EF database entity models, prevent over-posting vulnerabilities, reduce network payload sizes by returning only required fields, and decouple database schemas from client view structures.

Q: What are Action Filters and how do they work?

Action Filters are custom attributes that execute code before or after a controller action method runs. They are useful for cross-cutting tasks like log audits, input validation validations, or response caching. They implement the `IActionFilter` or `IAsyncActionFilter` interface.

Q: How do you handle global exceptions in ASP.NET Core Web API?

We handle global exceptions using custom exception handling middleware or the `IExceptionHandler` interface (introduced in .NET 8). The middleware wraps the execution pipeline in a try-catch block, intercepts unhandled errors, logs details using Serilog, and serializes a clean, standardized JSON `ProblemDetails` response (RFC 7807) to the client.

SQL & Entity Framework Core Real Interview Qs

Q: What is the difference between WHERE and HAVING clauses in SQL?

- `WHERE` clause: Filters rows *before* aggregation (GROUP BY) takes place. It cannot contain aggregate functions. - `HAVING` clause: Filters groups *after* aggregation has been performed. It is used in conjunction with the `GROUP BY` clause and can contain aggregate functions (e.g. `HAVING COUNT(*) > 1`).

Q: Explain the difference between DELETE, TRUNCATE, and DROP statements.

- `DELETE`: A DML command. It removes specific rows based on a WHERE filter. It executes row-by-row, triggers table triggers, writes log records, and can be rolled back. - `TRUNCATE`: A DDL command. It removes all rows from a table by deallocating data pages. It is faster, uses minimal logging, cannot use a WHERE filter, resets identity values, and triggers no row triggers. - `DROP`: A DDL command. It completely deletes the entire table structure, indexes, permissions, and data rows permanently from the database schema.

Q: What is a SQL View and what are its advantages?

A View is a virtual table representing the result of a precompiled SQL query. It does not store physical data. Advantages: 1. Simplifies complex queries by wrapping multiple joins into a single virtual table. 2. Restricts data access by exposing only specific columns to user groups. 3. Decouples application queries from the physical table schemas.

Q: What are Magic Tables (inserted and deleted) in SQL?

Magic Tables are temporary tables managed by SQL Server that exist only within the context of database Triggers. They store snapshots of affected rows: - `inserted` table: Stores copies of rows added during INSERT or modified values after UPDATE. - `deleted` table: Stores copies of rows removed during DELETE or original values before UPDATE.

Q: Explain the difference between UNION and UNION ALL.

- `UNION`: Merges the result sets of two SELECT queries and automatically runs a distinct filter to remove duplicate rows. It is slower due to sorting and deduplication overhead. - `UNION ALL`: Merges the result sets of two SELECT queries, keeping all duplicate rows. It is much faster because it performs no data scanning or sorting.

Q: What is a Surrogate Key vs Natural Key?

- `Natural Key`: A database key formed from columns that naturally exist as part of the business data (e.g. Social Security Number or unique Email). - `Surrogate Key`: An artificial key generated by the database system (usually an auto-incremented ID or GUID) that has no business meaning and is used solely for primary key indexing.

Q: What is Entity Framework Core and how does it act as an ORM?

EF Core is an object-relational mapper (ORM) that enables .NET developers to interact with relational databases using C# classes instead of writing raw SQL. It maps database tables to C# classes, translates LINQ queries into SQL statements, executes queries, and tracks changes to update records.

Q: How do you handle database relationships in EF Core?

We define relationships using Fluent API configurations in the `OnModelCreating` method of the `DbContext`: - One-to-One: `modelBuilder.Entity().HasOne(a => a.B).WithOne(b => b.A).HasForeignKey(b => b.AId)` - One-to-Many: `modelBuilder.Entity().HasMany(p => p.Children).WithOne(c => c.Parent)` - Many-to-Many: `modelBuilder.Entity().HasMany(a => a.Bs).WithMany(b => b.As)` (EF Core creates the join table automatically).

Q: What is the DbContext lifecycle and why must it be short-lived?

The `DbContext` represents a unit of work. Its lifecycle should be short-lived, typically scoped to a single HTTP request in web APIs. It should never be declared as a singleton. If it is long-lived, the change tracker will cache thousands of entities in memory, causing memory leaks, and concurrent requests sharing the same context instance will throw thread-safety crashes.

Q: What are Shadow Properties in EF Core?

Shadow Properties are properties that are not defined in the C# entity class but are configured and tracked by EF Core in the database schema. They are useful for cross-cutting audits, such as storing `CreatedDate` or `LastUpdatedBy` columns, without cluttering the domain model.
